# ADR: Location placed-object raster assets — Phase 1 manifest (Option A)

## Status

Accepted — Phase 1 (sprite migration plan).

## Context

Placed objects need stable **`assetId`** resolution to PNG files plus **generated** trim metadata, without hand-maintained per-asset trim tables. Toolbar previews and in-map sprites must share the same contract.

## Decision

- **Single manifest (Option A):** `assets/system/locations/objects/location-objects.manifest.json`, generated by `scripts/location-objects-assets/generateLocationObjectsManifest.ts`.
- **Per asset:** `map` and `preview` slices (`intrinsicSize` + alpha **trim** rect). **`map` is `null`** for door/window raster assets (edge families still use vector geometry on the map in Phase 2); **`preview` is always set** for every PNG.
- **`assetId`:** Derived from filename stem: `kebab-case` → `snake_case` (e.g. `door-single-wood.png` → `door_single_wood`).
- **Registry mapping:** `variantToAssetId.json` maps `{ family, variant }` → `assetId` or `null` until art exists.

## Consequences

- Phase 2 implements **one** resolver module against this shape (no parallel manifest formats).
- **Not chosen:** Option B (split map/preview manifests) — can be revisited in Phase 7 if atlas optimization requires it.
